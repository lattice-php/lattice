Lattice renders with a small set of semantic design tokens — CSS custom properties prefixed
with `--lt-`. Theme it by calling `createTheme`/`injectTheme` with a partial `Theme` — set only the
colors and scalars you want to change, and Lattice merges them over its defaults, deriving readable
foregrounds and hover/active states for you. For direct control over the underlying host variables,
see [Advanced: host variables](#advanced-host-variables).

Tokens are global and shared across components. The **Style** tab on each component example lists
the tokens that example uses in its initial rendered state. Tokens that only appear in
interactive or overlay content — a dropdown panel, a dialog, a popover that opens on demand — are
not captured there; this page is the complete reference.

## Rebranding

Pass a partial theme to `createTheme` and inject it before your app renders. Set only what you
want to change — Lattice derives readable foregrounds and hover/active states for you.

```tsx
import { injectTheme, initializeTheme } from "@lattice-php/lattice";

injectTheme({
  colors: { primary: "oklch(0.55 0.2 265)" },
  radius: "0.75rem",
  fontSans: "'Geist', sans-serif",
});
initializeTheme();
```

Brand dark mode by adding a `dark` block; otherwise Lattice keeps its tuned dark defaults:

```tsx
injectTheme({
  colors: { primary: "oklch(0.55 0.2 265)" },
  dark: { colors: { primary: "oklch(0.7 0.18 265)" } },
});
```

<Info>
  `injectTheme` writes to a managed `<style id="lattice-theme">` tag — it's SSR-safe and idempotent,
  so calling it again (e.g. after a runtime theme change) replaces the previous CSS in place. Prefer
  `createTheme(theme)` directly if you want the CSS string to inline yourself, for example during
  server-side rendering.
</Info>

## Server-side theming (PHP)

For themes known at boot — a single brand, or one resolved per tenant — build a `Theme` in PHP and
register it with the `Lattice` facade, typically in a service provider's `boot()` method:

```php
use Lattice\Lattice\Facades\Lattice;
use Lattice\Lattice\Support\Theme\Theme;

Lattice::theme(
    Theme::make()
        ->colors(primary: '#6366f1')
        ->radius('0.75rem')
        ->dark(fn (Theme $theme): Theme => $theme->colors(primary: 'oklch(0.7 0.18 265)')),
);
```

`colors()` accepts named arguments for any token (`primary`, `secondary`, `danger`, …) and derives
readable foregrounds and hover/active states, same as the JS path. `dark()` takes a closure that
builds the dark-mode overrides from a fresh `Theme`. Pass a closure instead of a `Theme` to `Lattice::theme()`
to resolve it per request — for example per-tenant branding:

```php
Lattice::theme(fn (): Theme => Theme::make()->colors(primary: tenant()->brand_color));
```

Or configure a theme declaratively in `config/lattice.php` under `frontend.theme`, using the same
structured shape:

```php
'frontend' => [
    'theme' => [
        'colors' => ['primary' => '#6366f1'],
        'radius' => '0.75rem',
        'dark' => ['colors' => ['primary' => 'oklch(0.7 0.18 265)']],
    ],
],
```

A theme registered via `Lattice::theme()` takes precedence over the config value.

Render it by adding `@latticeTheme` to your root Blade layout's `<head>`:

```blade
<head>
    @latticeTheme
    @vite(['resources/css/app.css', 'resources/js/app.tsx'])
    @inertiaHead
</head>
```

It emits a `<style id="lattice-theme">` block server-side, so the correct theme is present on first
paint — no flash of the default theme before JS runs.

<Info>
  The PHP `Theme` path is for themes known server-side (static or per-tenant), with correct first
  paint. The JS `injectTheme` path above is for runtime switching — a user-facing theme picker, for
  example. Both write to the same `lattice-theme` style id, so a client `injectTheme` call cleanly
  overrides the server-rendered theme. Hex, `rgb()`, and `oklch()` colors are all accepted on either
  path and get derived foregrounds and hover/active states automatically.
</Info>

## Dark mode

Every token ships a light and a dark value (the **Light** and **Dark** columns below). Dark mode is
driven by a `dark` class on the `<html>` element: when it is present, the `.dark` host variables
take over and Lattice's tokens follow. You theme dark mode the same way as light — by defining the
host variables under `.dark`.

Lattice manages the class for you through its appearance helpers, exported from the package:

- **`initializeTheme()`** — call once before your app mounts (e.g. in your entry file). It reads the
  stored preference (`localStorage` key `appearance`, default `system`), applies the `dark` class and
  `color-scheme`, and keeps them in sync when the OS preference changes.
- **`useAppearance()`** — a hook returning `{ appearance, resolvedAppearance, updateAppearance }`.
  `appearance` is the chosen mode (`"light" | "dark" | "system"`), `resolvedAppearance` is the
  effective `"light" | "dark"` after resolving `system`, and `updateAppearance(mode)` persists the
  choice (to `localStorage` and a cookie) and applies it immediately. Build your own theme switcher
  on top of it.

```tsx
import { initializeTheme, useAppearance } from "@lattice-php/lattice";

initializeTheme();

function AppearanceToggle() {
  const { appearance, updateAppearance } = useAppearance();

  return (
    <button onClick={() => updateAppearance(appearance === "dark" ? "light" : "dark")}>
      Toggle theme
    </button>
  );
}
```

<Info>
  The preference is persisted to both `localStorage` and an `appearance` cookie. The cookie lets the
  server render the correct theme on first paint, avoiding a flash of the wrong mode.
</Info>

## Advanced: host variables

Every token resolves through a host variable with a built-in fallback, so you can theme Lattice
without `createTheme` by defining the host variables (the same names shadcn/ui uses) directly on
`:root` and `.dark`. For example, set `--primary` and Lattice's `--lt-primary` follows.

```css
:root {
  --primary: oklch(0.55 0.2 265);
  --radius: 0.75rem;
}

.dark {
  --primary: oklch(0.7 0.18 265);
}
```

This is the lower-level mechanism `createTheme` generates CSS for. Reach for it when you already
own a design-token pipeline emitting these variable names, or need finer control than `Theme`
exposes — otherwise prefer `createTheme`/`injectTheme`, which also derives foregrounds and
hover/active states for you.
