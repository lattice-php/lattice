Lattice's built-in components — the rich-editor toolbar, table pagination, filter controls, the bulk
bar, and the accessibility labels on menus, toasts, tabs, and selects — read their text through
[i18next](https://www.i18next.com/), with the English baked into each call site as a fallback. The
UI is fully readable with **zero configuration**, and you can translate it by pointing Lattice's
i18next instance at a translation backend.

The instance is isolated from your app's own i18next setup (if any), so wiring it up never touches
your application's translations.

## The `lattice` namespace

Every built-in string lives in a single i18next namespace, `lattice`, with nested keys like
`editor.bold` or `pagination.next`. One namespace keeps Lattice's keys from colliding with your
app's, and means a single language file — and a single route — translates everything Lattice
renders.

## Zero config

There is nothing to set up to render in English. Each call site supplies its own default
(`t("editor.bold", "Bold")`), and the instance initializes itself the first time a component reads a
string. Apps that never opt into a backend never bundle one.

## Translating the built-in strings

To serve translations, install [`bambamboole/laravel-i18next`](https://github.com/bambamboole/laravel-i18next),
which exposes the routes the frontend reads from:

```bash
composer require bambamboole/laravel-i18next
```

Its routes register automatically. Enable them, and serve the `lattice` namespace as nested i18next
JSON:

```php
// config/i18next.php
return [
    'routes' => ['enabled' => true],
    'namespaces' => true,
    'output' => 'nested',
    // …see the laravel-i18next docs for the full config.
];
```

Lattice reads two values from this config — whether the routes are serving
(`i18next.routes.enabled`) and whether missing keys are reported back
(`i18next.save_missing.enabled`) — and shares them to the frontend on every page.

### Wire the frontend

Lattice serializes an `i18n` block onto every page's Inertia props, under `lattice.i18n`. Pass it to
`configureI18n` once in your Inertia setup, before the first render:

```tsx
// resources/js/app.tsx
import { configureI18n, type I18nConfig } from "@lattice-php/lattice/i18n";

createInertiaApp({
  // …resolve, layout…
  setup({ el, App, props }) {
    if (!el) {
      return;
    }

    const shared = props.initialPage.props as { lattice?: { i18n?: I18nConfig } };
    void configureI18n(shared.lattice?.i18n);

    createRoot(el).render(<App {...props} />);
  },
});
```

When the backend reports i18n as enabled, `configureI18n` attaches i18next's HTTP backend and loads
translations from laravel-i18next's routes, overriding the inline English. When it's disabled — or
the prop is absent — the inline English stands. Importing `configureI18n` is the opt-in: apps that
never import it don't bundle the HTTP backend.

### Add a language file

Translations for the `lattice` namespace live in `lang/{locale}/lattice.php`, with the same nested
keys Lattice uses. Lattice ships English and German; add your own locales the same way:

```php
// lang/de/lattice.php
return [
    'editor' => [
        'bold' => 'Fett',
        'heading-1' => 'Überschrift 1',
    ],
    'pagination' => [
        'next' => 'Weiter',
        'showing' => 'Zeige :from-:to von :total',
    ],
    // …
];
```

Interpolated values use Laravel's `:placeholder` syntax; laravel-i18next serves them as i18next
`{{placeholder}}` tokens.

## Using translations in your own components

The same instance is available to your [custom fields and columns](/extending/overview/). Inside a
component, use the `useT` hook with your own namespace:

```tsx
import { useT } from "@lattice-php/lattice/i18n";

function SaveButton() {
  const { t } = useT("app"); // served from lang/{locale}/app.php

  return <button>{t("save", "Save")}</button>;
}
```

Passing `"lattice"` reads Lattice's namespace instead — the built-in chrome does exactly that.
Outside of a component — building a label in a helper, say — use `translate`:

```ts
import { translate } from "@lattice-php/lattice/i18n";

translate("app", "save", "Save");
```

Both take the English fallback inline, so they are safe to call before any translations have loaded.

## Language detection

The instance picks its initial language from the `<html lang="…">` attribute, falling back to `en`.
Set it server-side from the app locale and Lattice follows. To switch at runtime, call
`changeLanguage` on the exported instance:

```ts
import { i18n } from "@lattice-php/lattice/i18n";

void i18n.changeLanguage("de");
```

## Reference

### `enableBackend(options?)`

`configureI18n` calls this for you, but you can call it directly for full control over the backend.
It registers i18next's HTTP backend and is the opt-in import.

| Option          | Default                          | Purpose                                                                 |
| --------------- | -------------------------------- | ----------------------------------------------------------------------- |
| `loadPath`      | `/locales/{{lng}}/{{ns}}.json`   | Route translations are fetched from.                                    |
| `addPath`       | `/locales/add/{{lng}}/{{ns}}`    | Route missing keys are reported to.                                     |
| `saveMissing`   | `false`                          | Report keys with no translation back to the backend, which persists them. |
| `customHeaders` | —                                | Returns extra request headers, e.g. a CSRF token for the `saveMissing` POST. |

The defaults match laravel-i18next's namespaced routes; override them only behind a custom route
prefix.

### The shared `i18n` prop

The block Lattice serializes onto every page, mirrored from laravel-i18next's config:

```ts
type I18nConfig = {
  enabled: boolean; // are the translation routes serving?
  saveMissing: boolean; // should missing keys be reported back?
};
```

`configureI18n(config)` enables the backend only when `enabled` is true, and forwards `saveMissing`.
The load and add paths are fixed on the frontend, so they never travel in this prop.
